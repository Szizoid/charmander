# Charmander — Architecture

Popup-утилита для вставки специальных символов на Wayland.
Аналог PowerToys Switch Accent, написанный на Rust.

---

## Концепция работы

1. Пользователь биндит запуск `charmander` на удобное сочетание клавиш в своём WM/DE
2. При нажатии хоткея появляется overlay-окно поверх всех окон (как rofi/wofi)
3. В окне: строка поиска + прокручиваемый список символов из библиотеки
4. Пользователь вводит запрос (часть имени, тег, похожий символ) — список фильтруется
5. Выбор символа через Enter или клик → символ вставляется в активное окно → окно закрывается

---

## Стек технологий

| Задача                    | Решение                          | Почему                                                                 |
|---------------------------|----------------------------------|------------------------------------------------------------------------|
| GUI на Wayland            | `gtk4` + `gtk4-layer-shell`      | Зрелая библиотека, поддерживается Hyprland/Sway/river и GNOME         |
| Overlay поверх окон       | `wlr-layer-shell` через GTK4 LS  | Стандартный протокол Wayland для панелей, уведомлений, launchers       |
| Конфигурация              | TOML-файл + крейт `toml`         | Человекочитаемый формат, легко редактировать вручную                   |
| Сериализация/десериал.    | `serde` + `serde_derive`         | Де-факто стандарт в экосистеме Rust                                    |
| Вставка символа           | `wtype` / `wl-copy` / `ydotool` | Разные окружения поддерживают разные инструменты — настраивается       |
| Аргументы CLI             | `clap`                           | Стандарт для CLI в Rust; пригодится для флагов `--config`, `--version` |
| Fuzzy-поиск               | Своя реализация или `fuzzy-matcher` | Простой алгоритм сначала, потом можно улучшить                      |

### Примечание по Wayland-совместимости

`gtk4-layer-shell` работает с **wlroots-based** композиторами (Hyprland, Sway, river, labwc...)
и с **GNOME** (через поддержку layer-shell в gnome-shell начиная с 45+).
KDE Plasma поддерживает layer-shell в Plasma 6+.

## Формат конфигурации

Конфиг хранится в `~/.config/charmander/config.toml`.
При первом запуске приложение копирует туда `default.toml` из пакета.

```toml
[settings]
# Метод вставки символа: "wtype", "wl-copy", "ydotool"
output_method = "wtype"

# Максимальное количество символов в списке результатов
max_results = 50

[[characters]]
symbol = "à"
name = "a grave"
tags = ["a", "latin", "accent", "grave", "french"]

[[characters]]
symbol = "é"
name = "e acute"
tags = ["e", "latin", "accent", "acute", "french", "spanish"]

[[characters]]
symbol = "α"
name = "alpha"
tags = ["a", "greek", "math", "physics"]

[[characters]]
symbol = "≈"
name = "approximately equal"
tags = ["math", "approx", "approximately", "equal"]

[[characters]]
symbol = "→"
name = "right arrow"
tags = ["arrow", "right", "math"]

[[characters]]
symbol = "©"
name = "copyright"
tags = ["legal", "copyright", "symbol"]
```

### Структура записи символа

- `symbol` — сам символ (строка, может быть многосимвольной, например лигатуры)
- `name` — полное имя (используется в поиске)
- `tags` — список тегов (используется в поиске)

## Алгоритм поиска и ранжирования

Входная строка запроса матчится против:
1. Точное совпадение с тегом → наивысший приоритет матча
2. Тег **начинается** с запроса → высокий приоритет матча
3. Тег **содержит** запрос → средний приоритет матча
4. Имя символа содержит запрос → низкий приоритет матча
5. (Опционально в будущем) fuzzy-match по имени

**Итоговый порядок результатов** определяется составной оценкой:

```
score = match_priority * 1000 + usage_count
```

То есть: сначала ранжируем по качеству совпадения, внутри одного уровня — по частоте использования. Чаще используемый символ всплывает выше среди равнозначных по матчу.

При пустой строке поиска список показывает **все символы, отсортированные только по частоте использования** (история использований в роли "избранного").

## Поток данных

```
запуск charmander
       │
       ▼
  [config.rs]  загрузка ~/.config/charmander/config.toml
  [history.rs] загрузка ~/.local/share/charmander/history.toml
       │
       ▼
  [db.rs] построение индекса символов в памяти
       │
       ▼
  [ui/window.rs] создание GTK4 окна через layer-shell (overlay)
       │
  пользователь вводит текст в поиск
       │
       ▼
  [search.rs] фильтрация + ранжирование (match_priority * 1000 + usage_count)
       │
       ▼
  [ui/window.rs] обновление списка
       │
  пользователь выбирает символ
       │
       ├──▶ [history.rs] инкремент счётчика + сохранение history.toml
       │
       └──▶ [output.rs] вызов wtype/wl-copy/ydotool → символ в активном окне
                 │
                 ▼
           приложение завершается
```

## Модули: ответственности

### `main.rs`
- Парсинг аргументов CLI (`--config`, `--version`, `--help`)
- Инициализация GTK-приложения
- Склейка всех модулей вместе

### `config.rs`
- Определение структур `Config`, `CharacterEntry`, `Settings`
- Загрузка конфига из файла
- Создание дефолтного конфига при первом запуске

### `db.rs`
- Хранение загруженных символов в памяти
- Предоставление интерфейса для модуля поиска

### `search.rs`
- Функция `search(query: &str, db: &Db) -> Vec<&CharacterEntry>`
- Приоритизация результатов

### `history.rs`
- Хранит счётчики использований в `~/.local/share/charmander/history.toml`
- Структура: `HashMap<String, u32>` где ключ — `symbol` (сам символ)
- Методы: `get_count(symbol)`, `increment(symbol)`, `load()`, `save()`
- Почему отдельный файл от конфига: конфиг редактирует пользователь вручную, историю пишет приложение — их нельзя смешивать

### `output.rs`
- Enum `OutputMethod { Wtype, WlCopy, Ydotool }`
- Функция `insert_character(symbol: &str, method: OutputMethod)`
- Вызов внешней программы через `std::process::Command`

### `ui/window.rs`
- Создание GTK4 ApplicationWindow
- Применение layer-shell (overlay, поверх всех окон)
- Поле ввода поиска → связь с `search.rs`
- Список результатов → GTK4 ListView
- Обработка Enter/клика → вызов `output.rs`

## Этапы разработки

### Этап 1 — Ядро без UI
- Структуры данных (`Config`, `CharacterEntry`, `Db`)
- Загрузка TOML-конфига
- Модуль истории (`history.rs`): чтение/запись счётчиков
- Алгоритм поиска с ранжированием по частоте
- Покрыть тестами

### Этап 2 — Вставка символа
- Модуль `output.rs`
- Тест через CLI (`charmander --insert "→"`)

### Этап 3 — GUI
- GTK4 окно
- Layer shell (overlay)
- Поиск в реальном времени
- Вставка при выборе

### Этап 4 — Полировка
- Дефолтная библиотека символов
- README, лицензия (MIT или Apache 2.0?)
- PKGBUILD для AUR
- Документация конфига
